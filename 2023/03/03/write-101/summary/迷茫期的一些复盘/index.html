1.初衷

在职场中迷茫是常见的经历，为了激励自己继续前行，总结过往经历并进行优化是一个不错的做法。

关于软件测试的经历，带着问题去实践寻找答案：

1.1 目标与动机：最初选择这个行业的动机是什么？目标是什么？

有两句话一直警示着我，在此分享给大家

  • 如果建筑工人盖房子的方式跟程序员写程序一样，那第一只飞来的啄木鸟就将毁掉人类文明
  • 软件测试的价值在于将原本暴露在用户面前的缺陷，提前在内部发现并解决

然后基于很多项目或公司实际情况，不能说测试无用但收益甚微【小公司尤其明显】，存在的问题可能是多种多样的，

  • 一方面是企业对质量管控的重视程度
  • 一方面是测试人员本身技能不足都有关系。

1.2 技能与知识：掌握技能主要是为了解决实际问题

  • 比如常见的接口测试，需要掌握一些接口调试工具：postman、curl等
  • 手工测试进度很慢就需要引入自动化，比如掌握一门编程语言和发请求的库，针对业务场景结合完成测试
  • 对于性能测试则需要掌握压测工具，特定场景下工具无法解决问题，就多掌握一门高性能编程语言，本质还是在处理发起请求的问题
  • 在分布式系统中重点关注数据一致性的解决方案，从理解共识算法开始更容易发现应用层的错误，以及发现错误后如何推理并找出问题所在

技能终究是工具，掌握多样化的工具是为了在解决问题时提供解决方案，去做别人不可替代的事才有价值。

2.软测基本三阶段

2.1 入门、初级

  • 入门不难，周期短，一般计算机专业的学生对于软件开发有一定经验，通过三个月的自学也能达到岗位需求
  • 技术栈比较浅，特别是在当前市场环境下，不推荐入坑

2.2 中、高级

  • 目前市场上需求，至少需要熟悉自动化测试、性能测试，掌握一门编程语言，有测试平台的开发经验更佳

  • 普遍岗位也没有太深入，大部分还是api调用工程师，不过能有专职的自动化岗位还是不错的，主要有时间能沉下心去学习，在持续的学习中接触到更多的知识才会发现自己掌握的只是冰山一角

  • 很多高级测试也只是编写维护自动化而已，自动化能解决一些问题，也会带来问题

    1. 什么时候要自动化？如何自动化？自动化程度如何，全部还是部分
    2. 维护成本也不能忽视，为了测试程序招了比开发还水的团队维护着大量的测试代码，这本就是一个伪命题

  • 测试开发岗、对于大厂而言还有精力和能力去打磨高效的测试平台，其要求的能力并不低普遍对标开发水平。除大厂内部自研的系统外，目前并没有明星产品的推出能一统天下，各系统不同关注点都需要编码去实现。

    推荐两个平台关注下：

    • RunnerGo go 微服务开发技术栈，一款全栈式测试平台
    • MeterSphere java 技术栈，一站式开源持续测试平台

2.3 资深、专家、测试管理

  • 某特定场景下能提供完整解决方案，拿人钱财替人消灾
  • 代码审计，软件架构，优化方案，解决核心问题的能力提高了自然会成为所谓的专家
  • 有了架构师，所谓测试架构师存在的意义又是什么呢？

3.改变

在步入web3行业后，工作中发现在此领域中大家对于软件的质量尤其关注，特别是智能合约

测试经验在智能合约开发中是尤为重要的一部分，web3领域对于安全的缺口也很大

4.总结

4.1 要做软测知识汇总，以实际工作为主，做实践项目输出

主要帮助的人群是长期功能测试/自动化无法落地的同学提升竞争力

4.2 要学习web3并形成个人的知识体系，并培养自己在web3行业的竞争力

除了个人需要较强的自制力外，还需要有良好的成员组队，以交流促进学习并相互监督

4.3 选对路比盲目努力重要

首先要相信选择的路是有希望的，才能坚持学下去，目标先掌握rust，上山的路其实并不拥挤
