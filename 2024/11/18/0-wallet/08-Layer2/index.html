Layer2

概述

提升底层网络(L1)的扩展性，并利用L1的安全性，其交易数据必须以某种形式被底层区块链网络验证并确认。
侧链就不属于Layer2，因为侧链通常会拥有自己的共识机制和验证节点，因此拥有独立于底层区块链的安全机制。
无需牺牲去中心化水平或安全性就能快速执行交易并实现可扩展性。

诞生的背景

区块链存在不可能三角,无法同时兼顾可扩展性、安全性和去中心化三个维度
当前区块链三个核心功能: 执行交易，数据可用性以及达成共识

  • 执行交易: 处理并完成交易。衡量指标是区块链每秒可以完成的计算次数（其中包含交易数量）。 
  • 数据可用性: 网络中的节点和验证者需要储存交易、状态以及其他数据。衡量指标是标准存储单位，比如MB和GB等。
  • 达成共识: 节点和验证者需要针对网络状态和交易排序达成共识。衡量指标是去中心化水平和终局速度，或所有节点针对某一状态变更达成一致意见所需的时间。

L2在结算时向底层提交某种可验证的加密证明，以证明状态变更的真实性。
L1需要智能合约支持用于验证L2证明

  • 保存并释放资金转账至L2
  • 收到L2提交的证明，进行验证并解决分歧，最终确认交易有效

支付通道

两个或以上用户可以预先向通道中充值资金，然后在链下进行通证转账。
链下记账的方式，只需要在通道开启时约定使用资金，后续交易都不会将记录发送至链上，而是在双方都决定关闭通道时，将最终交易结果发送至链上结算。

Rollups

Optimistic rollups和zk-rollups可以在链下执行智能合约状态变更，并在链上进行验证，以此提升区块链的吞吐量并降低成本。
1.Rollups在链下执行交易，底层区块链只需计算轻量化的交易证明，验证网络活动并储存原始交易数据。 
2.Rollups将交易数据打包，并提交至区块链，链上gas费可以分摊到各个交易中。 
3.Rollups最少只需要一个诚实的验证节点，就可以向底层区块链证明交易的有效性，因此对验证节点的数量要求较低，而对节点硬件要求较高，同时不会对安全性有很大影响。

链下执行

L2网络可以代替底层区块链处理与其他用户或智能合约的交易。L1只需要计算提交至智能合约的证明，就可以验证L2网络中的活动(op-rollup的话，只有在有分歧时才需要验证)，并将未执行的原始交易数据作为calldata储存起来。

打包交易

rollups将原始交易数据打包成calldata,Rollup交易的验证方式与链上交易不同，交易数据被打包只是为了将数据储存在底层区块链上，验证节点或rollup参与者如果有需要的话随时可以重建L2网络的状态。
主链上的一个打包交易中存储了多笔rollup交易。可以有效降低交易成本,将数据压缩还能进一步降低交易执行成本。

验证网络

  • 单验证者:L2利用L1来保证安全性和去中心化，L2必须向L1提交证明，来证明其发起的状态变更是有效的。
  • 多验证者:L2打包完数据后需要自己先达成共识，在往L1提交验证，L1上的智能合约验证提交证明有效性

L2的证明类型

  • 错误性证明（fault proof）, op-rollups
  • 有效性证明（validity proof，也称“零知识证明”）, zk-rollups

支付通道签名消息

支付通道创建后，双方都必须对他们发起的每一笔交易附上签名，并存储一份对方的签名。提交账本至L1执行判决结果并将正确的金额打到各自L1的钱包中进行结算。
若出现作恶场景，可以对结算交易发起挑战并提交最新的账本副本，智能合约判断，作恶方将受到惩罚。

op-rollups的错误性证明

采用这种证明方式的Layer2网络会默认所有交易都是有效的。然而，网络会设置一个挑战期，任何参与者都可以发起挑战，并向智能合约提交证明，表明交易数据或状态变更是错误的。
当错误性证明发布后，rollup交易会部分或全部在链上重新计算一遍，最终的状态变更会跟原来的结果进行比较。如果重新计算的结果不一样，原来的结果将被认为是无效的，并被撤回。一旦证明无误，验证节点的行为就会作废。

zk-rollups的有效性证明

证明L2网络中的交易和计算，L1上的智能合约可以验证L2网络提交的有效证明，以批准状态变更，功能完备的zk-rollup将不允许任何错误的交易在底层区块链上结算，因为每一批交易都必须附上相应的有效性证明。

扩容方案分类

执行层扩容

执行层指执行交易和状态变更的计算层。交易执行包含验证交易有效性，执行链上逻辑并计算状态变更。
执行层扩容通常指每秒处理的交易量，主要问题是如何在每秒处理更多计算量，并同时无需大幅提高对全节点验证区块链交易的依赖。

存储层

节点维持和储存账本副本的存储层。按存储功能通常分为两类:

  • 历史数据: 包含所有的原始交易和区块数据，交易数据包含from，to，amount以及每笔交易的签名。区块数据包含某一个区块哦的交易列表和元数据，比如根哈希值，nonce以及前一个区块的哈希值等。历史数据通常无需快速访问，只需要至少有一个诚实节点就可以下载。
  • 全局状态: 全部数据的快照，智能合约可以对其进行读写，比如所有智能合约的账户余额以及变量。全局状态可以看作是区块链的数据库，需要对输入的交易进行验证。状态通常存储在树形结构中，节点可以轻松快速的访问并更改。

节点需要访问历史数据，用于与区块链同步，也需要访问全局状态以验证新区块并执行新的状态变更，随着账本和相关存储数据量不断变大，状态计算会越来越慢，需要花更多的时间并执行更多的计算才能对状态读写。如果出现了状态膨胀就得升级硬件。

如何让区块链既能处理并验证更多数据，又不用提高对节点的存储要求呢？本质就是数据能不能换个地方存？

共识层扩容

区块链共识层指去中心化网络中的节点对区块链当前状态达成一致协议的地方。
区块链共识层的设计原则通常是在最大程度上降低通信成本，以提升去中心化水平的上限，实现更强大的拜占庭容错机制，并缩短终局时间。

保证可预测性、稳定性以及准确性的前提下，如何提高速度并降低成本，进一步实现信任最小化。

执行层扩容

提升验证节点的硬件，纵向扩容

硬件要求越高，意味着每个验证节点的计算能力越强，在每秒中能执行更多的计算。

打造多链生态，横向扩容

多个独立的区块链或某一区块链生态的侧链来进行横向扩容。将某一生态中的交易计算量分散到多个独立的链上，每条链都有自己的区块生产者和执行能力。

优点: 每条链都可以实现特定需求，或者实现 Dapp-chain, 拥有自我主权还可以隔离安全风险
缺点: 多链生态需要每条链都不断发行原生通证来建立自己的安全机制，用户交互往往会分散在各个不同的区块链上，难以实现规模效应

执行层分片，实现横向扩容

每个分片其实就是一个区块链，分片并行执行，会有一条主链来保证所有分片同步。
在执行分片时，验证者池也会分散到各个分片上去执行交易。节点会定期随机轮转，因此不会总是执行/验证同一个分片上的交易。

优点: 执行层的分片会从同一个节点池抽取节点，因此无需在新的分片上建立安全机制。如果节点池规模够大，每个执行环境都可以达到同样的安全水平，节点一次只需要执行一个分片上的计算任务，无需提高节点硬件要求

缺点: 分片之间存在数据依赖，需要将数据同步到各个分片，在分片灵活性上有一定限制。
主链计算需求越来越大，每个分片上分配的节点数量存在上限。
分片之间共享安全模式，所有分片都可能会出现同样的安全漏铜。

通过模块化实现横向扩容

将区块链的基础架构划分成执行层、数据可用性层（DA）以及共识层。最主流的区块链模块化机制就是rollup。
rollup 负责将计算和状态变更转移到链下网络，并在链上存储交易数据，然后使用zk-rollup 或 op-rollup对链下计算的状态变更结果进行链上验证

优点: rollup 在链下执行计算和状态变更能更高效的同时并保留底层L1的安全性，共识层在验证执行层链下计算时任然基于原有的L1达成共识。节点只需要验证简洁证明并存储少部分交易数据即可。
rollup还可以设置熔断机制，以实现信任最小化。如果rollup网络无法正常运行，用户可以提取自己的加密资产并提交至底层区块链。
模块化网络可以分摊用户成本，L1验证zk-rollup证明需要一定固定支出，更多的用户共同分摊会较少每个用户的共识成本。
rollup还具有1/n的信任模式，即使只有一个诚实节点也能保障计算的准确性和稳健性。

缺点: 模块化其复杂度相比单独运行的区块链更复杂,升级风险需要来自rollup以外的治理干预,因此无法保障不可篡改性。

支付和状态通道

用户将加密资产锁定在一个多签智能合约中，在链上交换签过名的消息，消息代表了资产的所有权转让或状态变更。整个过程无需发起任何链上交易，只有在创建通道和关闭通道时才需要支付费用。

用户可以通过多签合约对每笔交易进行加密签名，以此来保障通道中的结算正确无误，每个签名都会有一个nonce智能合约通过nonce来验证交易顺序正确性

优点: 支付和状态通道可以实现实时且零成本的转账，几乎不存在延迟，并且还能实现小额付款。
缺点: 每个参与方都保持联网状态,以确保交易对手不会使用旧消息在链上进行结算。也就是表明watchtower要持续对通道进行监控并保证用户资金安全，支付通道还需要预先充值，大额支付会很麻烦并且资金效率会很低。
支付通道中很难明确物品的所有权问题，比如在dex的流动性池。

数据存储层扩容

以下是扩容区块链存储层常用的几种方案，有些方案会合并在一起，以更好的提供存储能力

区块链节点纵向扩容

与纵向扩容区块链执行层一样，纵向扩容区块链存储层也需要升级全节点的硬件。 

底层区块链上进行数据分片

数据分片将账本数据或重建账本所使用的数据分成不同分片，降低对每个节点的存储要求。

模块化区块链压缩数据链上存储

模块化区块链在链下执行计算任务，然后将交易数据和状态变更储存在链上,其他节点可以用这些数据来重建账本的当前或历史状态。
rollup将数据存储在链上之前会先在链下进行数据压缩。

优点: 将数据存储在其他网络上，可以减少对L1的数据存储成本，并且存储也是去中心化的,随着使用量增加而更好地实现扩容
缺点: 链上存储的成本远高于链下存储，模块化区块链在可扩展性方便不如中心化的存储方案。
压缩数据会损失一些对验证性不是至关重要的数据，用户可能无法基于剩余的数据对链上活动进行更加细致的分析

数据修剪

数据修剪可以允许区块链节点删除某个区块高度之前的历史数据，数据修剪通常与PoS检查点(checkpoint)同时使用，超过某个检查点的区块中的交易会被认定终局。

优点: 数据修剪会减少节点参与共识时需要存储或参考的数据量。由于历史数据已经得到了验证，因此可以被修剪，已缩小账本规模。
缺点: 数据修剪需要依赖交易平台或区块浏览器等来永久存储历史数据，以一直追溯到创世区块，至少需要有一个第三方存储数据帮助节点重建所有的历史状态。

共识层扩容

共识层扩容主要涉及出块速度、终局速度以及节点下线或恶意攻击的稳健性。不仅是为了提高速度，还为了提升准确性、稳定性以及安全性

提高执行和存储能力

在账本不断扩大的过程中让更多的节点能参与共识，防止现有节点离开网络从而产生影响。
执行能力和存储能力得到提升同时不会对全节点产生太大的影响，那么区块链就可以实现更高的出块速度和更大的区块空间，并且无需牺牲去中心特质

减少网络带宽使用

降低节点达成共识所需的通信成本(收发消息的成本)。
无需与所有节点通信(all-to-all投票)，而是在任意时间与部分节点通信(重复抽样投票)即可，通过多轮投票或通行机制及降低成本。

降低网络延迟

区块链共识机制通过多轮重复抽样或all-to-all投票来实现即时终局。
也有部分区块链则设置了检查点，并由验证者在一段时间后达成共识来保障安全。这意味着区块在通过检查点后即被认为终局，之后再也无法在协议内展开区块重组。
网络延迟和网络带宽之间必须要做出一定权衡

提高安全预算

通过提供流动性，发放通证奖励或由于区块空间供不应求而提高交易费收入。提高安全预算会为参与者带来更多的潜在收入，这会提升网络的去中心化水平。
