全链路压测方案与实施

一、全链路压测的关键技术

1. 流量染色

对于压测流量经过的每一个中间件和服务进行准确识别，区分压测流量与正常流量。

2. 数据隔离

  • 影子数据库：在生产环境中使用影子数据库隔离压测数据和生产数据，避免对生产环境造成影响。
  • 影子数据表：保障整个链路可追踪，确保数据出现错误时具备恢复能力。

3. 风险管控机制

建立风险熔断机制，一旦发现生产环境的线上压测对业务造成影响，通过规则或指标自动触发风险熔断。这包括控制施压机流量，或对生产系统损坏部分进行业务隔离。

4. 日志隔离

由于日志是业务分析的重要数据来源，如不做日志隔离可能会对BI决策造成影响。例如，压测过程中大量使用某地域流量访问生产环境，可能导致BI分析错误判断该地区业务增长，从而做出错误决策。因此需要区分存储正常生产流量和压测流量的日志。

二、压测流程

完整的压测流程一般包含以下几个步骤：

  1. 压测目标的制定
  2. 压测链路的梳理
  3. 压测环境的准备
  4. 压测数据的构造
  5. 发压测试
  6. 瓶颈定位及容量微调
  7. 压测总结和报告

三、压测指标

根据业务需求选择关注的指标，常用指标包括：

指标 说明
QPS Query Per Second，每秒处理的请求个数
TPS Transactions Per Second，每秒处理的事务数（TPS ≤ QPS）
RT Response Time，响应时间，等价于Latency
CPU使用率 出于节点宕机后负载均衡考虑，一般保持 < 75%
内存使用率 观察内存占用情况，关注是否有尖刺或泄露
Load指标 CPU负载，表示一段时间内CPU正在处理及等待处理的进程数
缓存命中率 能命中缓存层（Redis、Memcached等）的流量比例
数据库耗时 数据库性能指标，通常是业务瓶颈的关键点
网络带宽 网络传输能力是否成为瓶颈
错误率 接口响应错误率或错误日志量

注意：RT分为平均延时和Pct延时（分位数）。平均值不能真实反映服务响应延时，实际压测中一般参考Pct90、Pct99等指标。

科学的评估方法应将指标、成功率、流量三者关联，例如：99%的响应在500毫秒内返回，且成功率为99.99%。

四、测试思路

  1. 响应时间约束下的最大吞吐量：在限定响应时间的条件下，系统能达到的最高QPS/TPS
  2. 100%成功率下的最大吞吐量：不考虑响应时间长短，保证所有请求成功的最大处理量
  3. 容忍部分失败的最大吞吐量：允许一定失败率和慢响应的情况下（如95%成功率，95%请求响应时间为特定毫秒）的最大QPS
  4. 持续性能表现：考虑不同持续时间下（如10分钟vs.1小时）系统的性能表现和资源占用情况，包括CPU、内存、负载、句柄、线程数、IO、带宽等是否合理

五、目标预估

压测前需要明确性能预期，以下是常用的目标预估方法：

1. 历史监控数据

对已上线且有监控的接口，可查看历史数据找出峰值QPS和PCT99。

2. 业务类比预估

对于新接口或未监控接口，可通过与功能类似的接口历史数据进行类比。

3. 业务场景估算

基于业务规模进行估算，常用”8/2原则”（一天内80%的请求会在20%的时间内到达）：

响应时间（RT）默认参考值：

  • 单服务单表类接口：RT < 100ms
  • 较复杂接口：RT < 300ms
  • 大数据量或调用链较长的接口：RT < 1s

示例1：电商秒杀活动

预估同时有1000万人参与，假设总QPS为1000万。

  • 前端秒杀倒计时形式使请求有2s的打散
  • 网关层做了20%几率拒绝请求的策略

计算：下单接口总QPS = 1000万 ÷ 2 × (1 - 0.2) = 400万/s

示例2：电商全天低价抢购活动

根据8/2原则，预估午休（12-1点）和晚上下班后（7-10点）共4小时是流量高峰。
计算：接口峰值QPS = 活动全天接口PV ÷ (4 × 3600s)

4. 逐步摸索法

针对”三无接口”（无参考、无预估、无历史数据），采用逐步增加压力的方式，收集数据并分析，最终确定接口的最优处理能力。
